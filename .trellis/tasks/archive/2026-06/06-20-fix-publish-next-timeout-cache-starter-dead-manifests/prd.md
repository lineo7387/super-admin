# Fix publish-next timeout cache and starter dead manifests

## Goal

Fix two confirmed release/starter quality issues: make the manual `publish-next` workflow less likely to hang indefinitely and faster/more stable on dependency installation, and prevent unregistered module manifest files from leaking into projects generated by `create-super-admin`.

## What I already know

- `publish-next.yml` currently gives `Run release check` a 15 minute timeout, but the actual `Publish selected packages` step has no step timeout and the job has no overall timeout.
- `publish-next.yml` uses `actions/setup-node` without pnpm cache settings, then runs `pnpm install --frozen-lockfile`.
- Prior Trellis records show a previous publish run fixed `pnpm: command not found`, a generated starter startup smoke hang, and local publish spec parsing. They do not record an intentional decision to leave publish steps without a timeout or dependency cache.
- `create-super-admin` generated starters intentionally keep example module directories such as dashboard, workbench, users, access, examples, ui-kit, and auth.
- The leak is narrower: standalone `access.manifest.ts`, `dashboard.manifest.ts`, `users.manifest.ts`, and `workbench.manifest.ts` are copied into generated starters even though current `module-registry.ts` only registers `examplesManifest` and `uiKitManifest`.
- A temporary generation check confirmed the default generated starter includes those four unregistered standalone manifests.

## Requirements

- Add pnpm dependency caching to the `Publish next` workflow through `actions/setup-node` using the repository lockfile.
- Add a bounded timeout to the actual `Publish selected packages` step.
- Add a reasonable overall job timeout so the manual workflow cannot run forever if an earlier or later step hangs.
- Keep the existing `Run release check` timeout in place unless implementation discovers a concrete reason to adjust it.
- Filter only the unregistered standalone module manifests from generated starter output:
  - `src/modules/access/access.manifest.ts`
  - `src/modules/dashboard/dashboard.manifest.ts`
  - `src/modules/users/users.manifest.ts`
  - `src/modules/workbench/workbench.manifest.ts`
- Preserve the actual example pages, query composables, types, mock-backed data flow, and registered `examples.manifest.ts` / `ui-kit.manifest.ts`.
- Apply the manifest filter to both source-root generation and the built runtime template path so local tests and published CLI behavior stay aligned.
- Add or update tests/validation so future generated starters fail if the four unregistered manifests return.

## Acceptance Criteria

- [ ] `.github/workflows/publish-next.yml` configures pnpm cache with `cache: pnpm` and `cache-dependency-path: pnpm-lock.yaml`.
- [ ] `.github/workflows/publish-next.yml` has a job-level timeout and a step-level timeout on `Publish selected packages`.
- [ ] Default `create-super-admin` output no longer contains standalone `access`, `dashboard`, `users`, or `workbench` manifest files.
- [ ] Generated output still contains `src/modules/examples/examples.manifest.ts` and `src/modules/ui-kit/ui-kit.manifest.ts`.
- [ ] Generated output still contains the example module pages/data files needed by `examples.manifest.ts`.
- [ ] Relevant CLI starter tests cover the removed manifests and still pass.
- [ ] Maintainer starter validation passes.

## Definition of Done

- Relevant tests pass, at minimum `pnpm --filter create-super-admin test`.
- Starter validation passes, preferably `pnpm validate:starter` unless runtime/network constraints block it.
- No npm registry-mutating commands are run.
- No generated starter behavior changes beyond removing the four dead manifest files.
- If implementation discovers a durable new convention, bring it back for spec-update review before finish-work.

## Technical Approach

- Update the publish workflow defensively: cache pnpm dependencies through `setup-node`, add a job timeout, and add a publish-step timeout.
- Introduce a single shared-looking manifest skip list in the CLI source generation path, then mirror the same filter in `scripts/build-cli-template.mjs` because the published CLI runs from `packages/cli/dist/starter-template/admin`.
- Extend `packages/cli/src/generate-starter.test.mjs` so the default generation test asserts the four dead manifests are absent and the two active manifests remain present.
- If there is an existing generated starter validator hook that naturally fits this rule, add a static validation check there too; otherwise keep the protection in CLI tests to avoid over-expanding scope.

## Decision (ADR-lite)

**Context**: The starter intentionally ships example modules, but current routing has folded several examples under `examples.manifest.ts`. Leaving the old standalone manifests in generated projects creates confusing dead code for users.

**Decision**: Keep the example module source directories, but exclude the unregistered standalone manifest files from generated output.

**Consequences**: Users still get copyable example pages and data patterns, while generated starter structure better matches actual route registration. Future first-class module generation can reintroduce manifest files deliberately when they are wired into the registry.

## Out of Scope

- Do not delete dashboard, workbench, users, access, examples, ui-kit, or auth example pages from generated starters.
- Do not change npm release channel policy, publish order, Trusted Publishing setup, or dist-tag behavior.
- Do not run `npm publish`, `npm dist-tag`, `npm trust`, or other registry-mutating commands.
- Do not add new CLI commands for module generation or example removal.
- Do not refactor the module registry architecture.
