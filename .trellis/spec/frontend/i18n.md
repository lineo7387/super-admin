# Internationalization Guidelines

## Purpose

Super Admin should become Chinese-first before CLI generation. The default admin app and generated projects should use `zh-CN` user-facing copy, with `en-US` available as an optional locale later.

This is part of release readiness, not a reason to delay the frontend-first scaffold or require a backend, CLI, generated schema, or provider.

## Locale Contract

- Default locale: `zh-CN`.
- Optional locale: `en-US`.
- Generated projects should default to `zh-CN` until a future CLI option explicitly selects another locale.
- New user-facing text should have a default Chinese message.
- Existing hard-coded English UI text should be migrated during the i18n phase before CLI generation begins.

## Message Organization

Use a central app-local message area first:

```text
apps/admin/src/i18n/
  index.ts
  locales/
    zh-CN.ts
    en-US.ts
```

Keep message keys namespaced by feature or surface, for example:

```text
shell.account.logout
auth.login.title
modules.users.empty.title
uiKit.buttons.primary
```

Do not put locale catalogs in `packages/ui`, `packages/core`, or `packages/theme` unless a later task defines package-level i18n ownership. Shared packages should stay reusable and avoid app-specific business copy.

## Migration Order

Move high-frequency user-facing text first:

1. Shell, navigation, account menu, preferences, and layout controls.
2. Auth login/register flows.
3. Users, access, dashboard, and workbench example modules.
4. UI Kit page titles, labels, state copy, and validation text.

Avoid mixing transport or API adapter changes into text migration unless a user-facing error message needs normalization.

## Rules

- Do not add new English-only user-facing copy.
- Do not translate internal identifiers, route names, test IDs, or API field names.
- Do not make i18n require the optional reference backend, CLI, CodeGraph, or generated schema.
- Keep text lookup available from Vue composition surfaces and small helpers; pages still follow the normal data path through query composables and API adapters.
- When changing the meaning of a message, update both `zh-CN` and `en-US` if both exist.
- Keep validation utilities pure by accepting an optional `MessageTranslator`; default to the app translator only inside `apps/admin`.
- Do not put app i18n imports in shared UI packages. Shared primitives should accept label props when app-specific localized copy is needed.

## Runtime Convention

Use Vue I18n in Composition API mode for the admin app:

```ts
createI18n({
  fallbackLocale: 'zh-CN',
  legacy: false,
  locale: 'zh-CN',
  messages
})
```

Components should call `useI18n()` and pass `t` into module helpers that need user-facing messages:

```ts
const { t } = useI18n()
const errors = validateUserInput(form, t)
```

Pure module helpers may default to the app-level translator for ordinary calls, but tests should also cover injected `en-US` translators for migrated validation messages.

## Tests And Verification

When i18n code exists, test at least:

- the default locale resolves to `zh-CN`
- missing keys are caught or reported in a predictable way
- switching to `en-US` does not break shell/auth/module rendering
- generated projects still start with Chinese default copy
